Phần 1 — Session & Quản lý trạng thái
1 Session trong JSP là gì? Nó được tạo ra khi nào và lưu ở đâu (client hay server)? Session▸

Session là vùng nhớ trên server gắn với từng user riêng biệt, được nhận diện qua JSESSIONID cookie gửi về browser. Được tạo lần đầu khi gọi request.getSession() hoặc khi trang JSP đầu tiên được truy cập. Dữ liệu lưu server-side — client chỉ giữ ID.

Hỏi thêm: nếu user tắt cookie thì session sẽ thế nào? → URL rewriting (;jsessionid=xxx).
2 Code lấy session.getAttribute("loginError") rồi gọi session.removeAttribute("loginError") ngay lập tức — nếu không removeAttribute thì điều gì xảy ra? Session▸

Thông báo lỗi sẽ hiển thị mãi mỗi lần F5 lại trang — dù user chưa làm gì sai thêm. Pattern "read then remove" gọi là flash attribute — chỉ hiển thị một lần rồi biến mất. Đây là cách xử lý thông báo sau redirect đúng chuẩn.

3 Giỏ hàng được lưu kiểu Map<Integer, Object[]> trong session. Object[] chứa gì? Tại sao không lưu trực tiếp số lượng vào một Map khác? Session▸

Object[0] là object products, Object[1] là int số lượng. Gộp cả 2 vào một entry cho phép duyệt giỏ hàng chỉ bằng 1 vòng lặp, lấy được cả thông tin sản phẩm lẫn số lượng cùng lúc. Nếu tách ra 2 Map phải lookup 2 lần.

Nhược điểm: Object[] không type-safe, phải cast thủ công, dễ ClassCastException. Nên dùng class CartItem(products, int qty) thay thế.
4 Tại sao giỏ hàng dùng LinkedHashMap thay vì HashMap thông thường? Session▸

LinkedHashMap duy trì thứ tự chèn (insertion order) — sản phẩm thêm vào trước hiển thị trước. HashMap không đảm bảo thứ tự — mỗi lần load giỏ hàng thứ tự sản phẩm có thể thay đổi ngẫu nhiên, gây trải nghiệm xấu.

5 Code: Map cart = (Map) session.getAttribute("cart"); if (cart == null) cart = new LinkedHashMap(); — tại sao cần kiểm tra null trước khi tạo mới? Session▸

Lần đầu user vào trang giỏ hàng chưa có gì trong session → getAttribute trả về null. Nếu không check null và dùng trực tiếp → NullPointerException khi gọi cart.values(). Tuy nhiên cart mới tạo này không được lưu lại vào session — thiếu session.setAttribute("cart", cart).

Đây là bug tiềm ẩn: cart tạo local không được persist vào session.
6 Session timeout mặc định trong Tomcat là bao nhiêu phút? Nếu session hết hạn, giỏ hàng user bị sao? Cách khắc phục? Session▸

Mặc định 30 phút. Giỏ hàng mất hoàn toàn vì lưu trong session. Khắc phục: (1) Tăng timeout trong web.xml: <session-timeout>60</session-timeout>. (2) Lưu giỏ hàng vào database khi user đăng nhập, restore khi login lại. (3) Dùng localStorage để cache client-side.

7 session.getAttribute("cartCount") trả về kiểu gì? Tại sao JSP viết session.getAttribute("cartCount") != null ? session.getAttribute("cartCount") : 0? Session▸

Trả về Object (thực chất là Integer). Nếu user chưa thêm gì vào giỏ, cartCount chưa được set → null. Toán tử ternary ngăn hiển thị "null" ra màn hình, thay bằng 0. JSP sẽ gọi .toString() tự động khi render.

8 User A đăng nhập, thêm vào giỏ 3 sản phẩm, rồi đăng xuất. User B đăng nhập cùng máy. User B có thấy giỏ hàng của user A không? Tại sao? Session▸

Phụ thuộc vào code logout. Nếu logout chỉ xóa user attribute mà không session.invalidate(), session cũ vẫn còn và cart vẫn còn → User B sẽ thấy giỏ hàng của A. Đúng ra phải gọi session.invalidate() khi logout để hủy toàn bộ session.

Đây là lỗ hổng bảo mật thực tế — giảng viên rất hay hỏi về vấn đề này.
9 Khác nhau giữa session.setAttribute() và request.setAttribute() — cái nào tồn tại lâu hơn? Session▸

request.setAttribute(): chỉ tồn tại trong 1 request-response cycle (một lần forward). Khi redirect hoặc request mới → mất. session.setAttribute(): tồn tại xuyên suốt nhiều request cho đến khi session timeout hoặc invalidate. Project dùng cả 2: product list qua request, user login qua session.

10 Sau khi đăng nhập thành công, code set session.setAttribute("loginSuccess", "Đăng nhập thành công!") rồi redirect. Tại sao cần lưu thông báo vào session thay vì request? Session▸

Vì dùng response.sendRedirect() — redirect tạo request mới từ browser, request cũ (cùng attributes) bị hủy. Chỉ session mới sống sót qua redirect. Đây là pattern Post/Redirect/Get (PRG) — submit form → redirect → GET page mới, tránh F5 gửi lại form.

Phần 2 — JSP & Java Logic
11 "admin".equals(session.getAttribute("role")) — tại sao đặt chuỗi literal ở trước, không viết ngược lại? JSP/Java▸

Nếu session.getAttribute("role") trả về null (chưa đăng nhập), gọi null.equals("admin") ném NullPointerException. Đặt literal trước: "admin".equals(null) → trả về false an toàn. Gọi là Yoda condition — phòng thủ null.

12 NumberFormat.getCurrencyInstance(new Locale("vi", "VN")) — output của 1500000 trông như thế nào? Nếu không format thì hiển thị ra sao? JSP/Java▸

Output: 1.500.000 ₫. Dấu chấm phân cách hàng nghìn theo chuẩn Việt Nam, ký hiệu ₫ tự động. Không format: hiển thị 1500000.0 hoặc 1500000 — thiếu dấu phân cách, không có ký hiệu tiền tệ, trông rất nghiệp dư.

13 Tính giá gốc từ giá khuyến mãi: double origPrice = p.getPrice() * 100.0 / (100.0 - p.getDiscount_price()) — giải thích công thức này. JSP/Java▸

Nếu discount_price = 20 (giảm 20%), thì giá hiện tại = giá gốc × 80%. Đảo lại: giá gốc = giá hiện tại × 100 / 80. Code: currentPrice × 100.0 / (100.0 - 20). Dùng 100.0 (double) thay 100 (int) để tránh integer division.

14 Math.round(origPrice / 1000) * 1000 — đoạn này làm gì? Tại sao cần làm tròn đến nghìn? JSP/Java▸

Làm tròn giá tiền đến đơn vị 1000 đồng. Ví dụ: 1.875.000 → 1.875.000 (đã chẵn). 1.876.543 → 1.877.000. Giá nội thất thường được niêm yết theo mức nghìn — tránh hiện giá lẻ như "1.876.543 ₫" trông không chuyên nghiệp.

15 Vòng lặp trong JSP: <% for (products p : productList) { %> ... <% } %> — scriptlet <% %> có vấn đề gì? Cách hiện đại hơn là gì? JSP/Java▸

Scriptlet trộn Java code vào HTML → khó đọc, khó maintain, không tách biệt logic và view. Cách hiện đại: JSTL (JSP Standard Tag Library) với <c:forEach items="${products}" var="p">. Hoặc chuyển sang template engine như Thymeleaf. Tuy nhiên, với project học phạm vi nhỏ, scriptlet vẫn chấp nhận được.

16 if (productList == null || productList.isEmpty()) — tại sao cần kiểm tra cả null VÀ isEmpty()? Chỉ một cái không đủ sao? JSP/Java▸

Chỉ == null: nếu Servlet set attribute là danh sách rỗng (không phải null) → không vào nhánh if → vòng lặp chạy 0 lần, không hiển thị gì. Chỉ isEmpty(): nếu null → NullPointerException. Kết hợp cả 2 an toàn tuyệt đối.

17 Trang product_detail lấy sản phẩm bằng request.getAttribute("product"). Nếu p == null, code hiển thị thông báo "Không tìm thấy". Điều này xảy ra trong trường hợp nào? JSP/Java▸

(1) User truy cập URL với ?id= không tồn tại trong DB. (2) Servlet quên request.setAttribute("product", p). (3) ID không phải số nguyên → Servlet ném exception trước khi set attribute. (4) Sản phẩm bị xóa trong DB nhưng user vẫn giữ URL cũ.

18 p.getProduct_name().replace("'", "") trong onclick của button — tại sao phải replace dấu nháy đơn? JSP/Java▸

Vì onclick dùng: onclick="addToCart(1, 'Ghế Gỗ Sồi')". Nếu tên có dấu nháy đơn như "Ghế Gỗ Sồi 'Vintage'" → HTML thành: addToCart(1, 'Ghế Gỗ Sồi 'Vintage'') → JavaScript syntax error, hàm không gọi được. Replace trước để thoát ký tự đặc biệt.

Giải pháp đúng hơn: dùng encodeURIComponent hoặc data-attribute thay vì nhúng string vào onclick.
19 Tính phí ship: double shipping = subtotal >= 500000 ? 0 : 30000 — logic này đúng chưa? Nếu subtotal = 499999 thì ship bao nhiêu? JSP/Java▸

Logic đúng. subtotal = 499999 → 499999 < 500000 → shipping = 30.000đ. subtotal = 500.000 → 500000 >= 500000 → shipping = 0. Threshold chính xác tại 500.000đ. Ngoài ra code cũng hiển thị "Mua thêm N đồng để miễn phí ship" khi shipping > 0.

20 Tính total = subtotal + shipping trên JSP — nhưng khi user cập nhật số lượng, total hiển thị có tự cập nhật không? Tại sao? Bẫy▸

Không tự cập nhật. subtotal và total được tính một lần khi JSP render — đây là giá trị tĩnh trong HTML. Khi user thay đổi số lượng qua JS (gọi UpdateCart), code dùng location.reload() sau 800ms để tải lại trang, render lại tổng tiền mới từ server.

21 java.net.URLEncoder.encode(p.getProduct_name(), "UTF-8") trong thuộc tính onerror của ảnh — tại sao cần encode? JSP/Java▸

Tên sản phẩm có thể chứa khoảng trắng, dấu tiếng Việt — không hợp lệ trong URL. URLEncoder chuyển "Ghế Gỗ Sồi" → "Gh%E1%BA%BF+G%E1%BB%97+S%E1%BB%93i". placehold.co dùng text trong URL để render chữ lên ảnh placeholder. Không encode → URL sai → ảnh placeholder không load.

22 Trong JSP, <%= %> và <% out.print() %> có khác nhau không? JSP/Java▸

Tương đương nhau — <%= expr %> là shorthand của <% out.print(expr); %>. JSP engine biên dịch cả 2 ra cùng một Java code. Tuy nhiên <%= %> gọn hơn, được dùng phổ biến hơn trong code của project.

23 <title><%= p != null ? p.getProduct_name() : "Chi tiết sản phẩm" %> - Decor Luxury</title> — tại sao cần kiểm tra null trong thẻ title? JSP/Java▸

Thẻ <head> được render trước phần body. Nếu p null và gọi p.getProduct_name() trong title → NullPointerException ngay tại đây, trang lỗi 500 trước khi hiển thị được thông báo "Không tìm thấy sản phẩm" ở body.

24 Sidebar danh mục dùng onclick="location.href='...'" trên <li> thay vì thẻ <a>. Vấn đề accessibility nào phát sinh? Bẫy▸

Thẻ <li> không focusable bằng bàn phím Tab, không được screen reader nhận diện là link. User dùng bàn phím hoặc screen reader không thể navigate qua danh mục. Nên dùng <a href="..."> thật sự bên trong <li> để đúng semantic HTML.

25 Dropdown nav lấy categories từ request.getAttribute("categories"). Ai set attribute này? Có trường hợp nào navCats bị null không? JSP/Java▸

Servlet (HomeServlet/ProductServlet) set attribute trước khi forward đến JSP. navCats null khi: (1) Servlet quên set. (2) DB truy vấn lỗi, trả về null. (3) Trang được truy cập trực tiếp không qua Servlet (gõ URL .jsp trực tiếp). Trong cart.jsp có fallback hardcode các phòng khi null — home_page.jsp thì không có fallback.

Phần 3 — JavaScript Logic
26 Hàm showToast(msg, isError = false) — isError = false là gì trong JavaScript? Nếu gọi showToast("Lỗi!") không truyền tham số thứ 2 thì sao? JavaScript▸

isError = false là default parameter (ES6+). Nếu gọi không có tham số 2, isError tự động nhận giá trị false. Toast hiện màu xanh (success). Chỉ khi gọi showToast("msg", true) mới hiện màu đỏ (error).

27 window.addEventListener('DOMContentLoaded', function() { showToast(...) }) — tại sao cần DOMContentLoaded? Nếu gọi showToast() trực tiếp trong <script> không được sao? JavaScript▸

showToast() dùng document.getElementById('toast'). Nếu script chạy trước khi HTML được parse xong, element toast chưa tồn tại → getElementById trả về null → null.classList.add() → TypeError. DOMContentLoaded đảm bảo DOM đã sẵn sàng trước khi chạy.

28 const urlParams = new URLSearchParams(window.location.search) dùng để làm gì trong code? Cho ví dụ URL kích hoạt điều này. JavaScript▸

Parse query string từ URL. Nếu URL là /home?openModal=login, urlParams.get("openModal") trả về "login" → tự động mở modal đăng nhập. Dùng trong trường hợp user bấm "Thêm vào giỏ" mà chưa đăng nhập → Servlet redirect về home với ?openModal=login.

29 Hàm toggleChat() kiểm tra trạng thái bằng chat.style.opacity === '1' thay vì biến boolean. Rủi ro gì có thể xảy ra? JavaScript▸

Nếu opacity chưa được set bằng JS (lần đầu load), chat.style.opacity trả về chuỗi rỗng "" dù CSS có set opacity. JS chỉ đọc được inline style, không đọc CSS file. → So sánh "" === '1' → false → xử lý như đang đóng, logic có thể sai. Nên dùng classList.toggle('open') thay thế.

30 localStorage.removeItem("chatHistory") được gọi khi đóng chat. Nhưng project không thấy dùng localStorage.setItem ở đâu — dòng này có tác dụng gì không? Bẫy▸

Không có tác dụng gì — xóa key không tồn tại thì không lỗi nhưng cũng không làm gì. Đây là code thừa (dead code), có thể là tính năng lưu lịch sử chat đã bị bỏ dở giữa chừng. Nên xóa để code gọn.

31 Hàm updateCartQty(productId, delta) clamp giá trị: if (val < 1) val = 1; if (val > 99) val = 99 — tại sao cần clamp trên cả client lẫn server? JavaScript▸

Client-side clamp chỉ là UX — ngăn số âm hiển thị trên UI. User vẫn có thể gửi request thẳng với số lượng tùy ý (bypass JS bằng Postman hoặc sửa URL). Server phải validate lại: Math.max(1, Math.min(99, qty)). Never trust client input.

32 Hàm applyCoupon() hardcode danh sách mã trong JS: const validCodes = { 'GIAM10': 10, ... }. Đây có phải vấn đề bảo mật không? Bảo mật▸

Rất nguy hiểm — mọi user có thể xem mã giảm giá bằng cách mở DevTools → Sources → xem JavaScript. Thực tế giảm giá phải validate trên server, không để logic trên client. Hơn nữa, toast "Áp dụng mã thành công" chỉ là thông báo — không thực sự trừ tiền.

Đây là lỗ hổng bảo mật + bug logic nghiêm trọng nhất trong toàn bộ project.
33 Hàm localReply(msg) dùng .normalize("NFD").replace(/[\u0300-\u036f]/g, "") — đoạn này làm gì? JavaScript▸

Xóa dấu tiếng Việt để so sánh không phân biệt dấu. normalize("NFD"): tách ký tự có dấu thành ký tự gốc + combining mark. replace([\u0300-\u036f]): xóa toàn bộ combining marks. "xin chào" → "xin chao". Vậy regex /xin chao|hello/ match được cả có dấu và không dấu.

34 Hàm addMessage() dùng div.innerHTML = formatted để render tin nhắn chat — rủi ro bảo mật gì có thể xảy ra? Bảo mật▸

XSS (Cross-Site Scripting). Nếu bot trả về <script>alert('hack')</script>, innerHTML sẽ thực thi script đó. Cách an toàn: dùng textContent thay innerHTML cho text thuần, hoặc sanitize HTML trước. Code hiện tại chỉ convert markdown cơ bản nhưng không escape HTML tags.

35 Hàm changeQty(delta) trong product detail — nếu user gõ tay 0 vào input number thì sao? Code có bảo vệ không? JavaScript▸

Input có thuộc tính min="1" ngăn browser UI nhập số < 1. Nhưng changeQty chỉ dùng parseInt(input.value) + delta không check min/max — nếu user bypass min attribute, có thể gửi qty=0 hoặc âm. Cần thêm: if (newVal < 1) newVal = 1;.

36 Hàm switchModal(closeId, openId) — tại sao không đóng tất cả modal rồi mở cái mới, mà chỉ đóng modal cụ thể? JavaScript▸

Project chỉ có 2 modal (login, register) và chúng mutually exclusive — không bao giờ cả 2 cùng mở. Truyền closeId cụ thể đủ dùng. Nếu sau này thêm nhiều modal, nên refactor: document.querySelectorAll('.modal').forEach(m => m.style.display='none') rồi mở cái cần.

37 Badge giỏ hàng cập nhật client-side: badge.textContent = parseInt(badge.textContent || 0) + 1 — vấn đề gì khi user thêm cùng sản phẩm 2 lần? Bẫy▸

Giỏ hàng thực tế cộng dồn số lượng sản phẩm đó (qty tăng), không thêm dòng mới — tức chỉ 1 item trong cart nhưng badge có thể tăng 2 lần. Badge count nên phản ánh tổng số loại sản phẩm hoặc tổng số lượng, nhưng cả 2 đều không đúng khi chỉ cộng +1 mỗi lần click.

38 Scroll top button: window.addEventListener('scroll', () => { if (window.scrollY > 300) ... }) — event listener này có vấn đề hiệu năng không? JavaScript▸

Scroll event bắn rất nhiều lần khi user scroll — có thể hàng chục lần/giây. Mỗi lần đều chạy callback. Cách tối ưu: dùng throttle (giới hạn tần suất) hoặc requestAnimationFrame. Với code đơn giản như này (chỉ add/remove class) thì ảnh hưởng không đáng kể nhưng đây là nguyên tắc cần biết.

39 Hàm switchTab(tabId) — nếu không tìm thấy element với ID đó thì document.getElementById trả về gì? Code có bị lỗi không? JavaScript▸

Trả về null. Sau đó null.classList.remove('active') → TypeError: Cannot read properties of null. Code không có try/catch bảo vệ → lỗi console, tab không switch. Nên thêm: if (panel) panel.classList.add('active');

40 let isSending = false được dùng để làm gì trong hàm sendMsg()? JavaScript▸

Flag ngăn user gửi nhiều tin nhắn liên tiếp khi đang chờ response (debounce/lock). Khi đang gửi: isSending = true, hàm return sớm nếu bị gọi lại. Sau 1.5s trong finally: isSending = false, mở khóa. Không có cơ chế này, user spam click "Gửi" sẽ flood server với nhiều request.

Phần 4 — Fetch API & AJAX
41 addToCart() dùng fetch(..., { method: 'GET' }) — tại sao thêm vào giỏ lại dùng GET thay vì POST? Có vi phạm HTTP semantics không? Fetch/AJAX▸

Có vi phạm semantics. GET chỉ dùng để đọc dữ liệu (idempotent). "Thêm vào giỏ" thay đổi trạng thái server → phải dùng POST. Dùng GET có thể bị browser/CDN cache, bị prefetch nhầm bởi trình duyệt, và có thể bị gọi nhiều lần nếu user refresh. Đây là lỗi thiết kế API.

42 Fetch chain: .then(res => { if (res.ok) {...} else {...} }).catch(() => {...}) — res.ok là gì? Khác gì so với .catch()? Fetch/AJAX▸

res.ok: true nếu HTTP status 200-299. Fetch không reject với HTTP 4xx/5xx — promise vẫn resolve nhưng res.ok = false. .catch() chỉ bắt lỗi network (mất mạng, DNS fail, CORS block). Cần cả 2: check res.ok cho HTTP error, .catch() cho network error.

43 UpdateCart trả về JSON, code dùng .then(res => res.json()). Nếu server trả về HTML (trang lỗi 500) thay vì JSON thì res.json() bị sao? Fetch/AJAX▸

res.json() cố parse HTML như JSON → throw SyntaxError: Unexpected token <. Nếu không có .catch(), lỗi không được xử lý. Code trong cart.jsp có .catch(() => { showToast('Đã cập nhật!'); location.reload() }) — tức là kể cả lỗi cũng reload, che giấu error thực sự.

44 Chat gọi: body: "message=" + encodeURIComponent(msg) với header Content-Type: application/x-www-form-urlencoded. Tại sao cần encodeURIComponent? Fetch/AJAX▸

URL form encoding yêu cầu ký tự đặc biệt phải được encode. Ví dụ: "Sofa & bàn" → nếu không encode, "&" bị hiểu là tách param, server đọc "message=Sofa " và "bàn" là param riêng. encodeURIComponent: "Sofa %26 b%C3%A0n" — server nhận đúng.

45 setTimeout(() => location.reload(), 800) sau khi update cart — tại sao cần delay 800ms? Reload ngay không được sao? Fetch/AJAX▸

Để user đọc được toast thông báo "Đã cập nhật số lượng!" trước khi trang reload. Reload ngay lập tức → toast vừa hiện đã mất, UX tệ. 800ms là đủ để user nhận ra feedback. Toast hiện trong 2800ms nhưng chỉ cần 800ms là đủ để user thấy.

46 Chat dùng fetch gọi /ChatServlet — nếu ChatServlet xử lý chậm (5 giây), UX có bị ảnh hưởng? Code xử lý thế nào? Fetch/AJAX▸

Trong 5 giây: nút Gửi bị disabled (opacity 0.5), typing indicator hiện ("Đang soạn tin..."), isSending = true ngăn gửi thêm. Đây là UX tốt — user biết đang chờ. Tuy nhiên không có timeout — nếu server treo vô hạn, user phải tự F5. Nên thêm AbortController với timeout.

47 Hàm quickSend(text) set chatInput.value = text rồi gọi sendMsg() — nhưng quick replies vẫn hiển thị sau khi gửi. Code nào ẩn quick replies? JavaScript▸

Code trong minimize(): nếu chatMessages.children.length > 3 thì ẩn quickReplies. Nhưng trong sendMsg() thông thường không có logic ẩn quick replies. Vậy quick replies luôn hiển thị trong suốt cuộc chat — có thể là bug UI, nên ẩn quick replies sau lần chat đầu tiên.

48 Markdown converter trong addMessage(): text.replace(/\[([^\]]+)\]\(([^)]+)\)/g, ...) — regex này match pattern gì? JavaScript▸

Match markdown link: [text](url). [^\]]+: bất kỳ ký tự nào trừ ] (phần text). [^)]+: bất kỳ ký tự nào trừ ) (phần URL). Thay bằng <a href="url">text</a>. Bot có thể trả về link markdown → hiển thị thành link HTML clickable.

Phần 5 — MVC & Kiến trúc Servlet
49 MVC trong project này gồm những gì? Ai đóng vai M, V, C? MVC▸

M (Model): class products, DAO classes — chứa dữ liệu và logic nghiệp vụ. V (View): JSP files (home_page.jsp, cart.jsp...) — chỉ hiển thị, không xử lý logic. C (Controller): Servlet (HomeServlet, AddToCartServlet...) — nhận request, gọi DAO, set attribute, forward đến JSP.

50 Tại sao không nên để code SQL trực tiếp trong JSP? Project có làm vậy không? MVC▸

Không nên vì vi phạm MVC, JSP trở nên khổng lồ và khó test, logic nghiệp vụ bị trộn vào View. Project không để SQL trong JSP — JSP chỉ nhận data từ request.getAttribute() do Servlet set. Logic DB nằm trong DAO layer. Đây là điểm tốt của kiến trúc.

51 request.getRequestDispatcher("home_page.jsp").forward(request, response) — forward khác response.sendRedirect() như thế nào? URL trên browser thay đổi không? MVC▸

forward: server-side, cùng request, URL không đổi trên browser (vẫn hiện /home), nhanh hơn vì không tạo request mới. redirect: server gửi 302 về browser, browser tạo request GET mới, URL đổi. Dùng forward khi muốn giữ attributes; redirect sau POST để tránh resubmit.

52 Cả login và register đều action="login" method="post" — một Servlet xử lý 2 chức năng. Servlet phân biệt bằng cách nào? MVC▸

Button submit có name="action" value="login" hoặc value="register". Servlet đọc request.getParameter("action"): nếu "login" → xử lý đăng nhập; nếu "register" → xử lý đăng ký; nếu "logout" → hủy session. Đây là Command Pattern đơn giản — một endpoint, nhiều lệnh.

53 Tại sao Servlet xử lý xong đăng nhập lại sendRedirect về trang chủ thay vì forward đến JSP? MVC▸

Pattern Post/Redirect/Get. Nếu forward sau POST, URL vẫn là /login. User F5 → browser hỏi "gửi lại form?" → đăng nhập 2 lần hoặc lỗi. Redirect về GET request mới, F5 chỉ GET trang chủ, không gửi lại form. Bắt buộc dùng redirect sau mọi POST quan trọng.

54 Servlet AddToCart nhận productId từ query string — làm sao Servlet lấy được product object để lưu vào giỏ? Phải gọi DAO không? MVC▸

Có — phải gọi products_DAO.getProductById(productId) để lấy object đầy đủ (tên, giá, ảnh...). Sau đó lưu vào Map cart trong session. Không thể chỉ lưu productId vì JSP cần render tên, giá, ảnh khi hiển thị giỏ hàng — không muốn query DB mỗi lần hiển thị.

55 Tại sao URL Servlet dùng ${pageContext.request.contextPath}/AddToCart thay vì /AddToCart hay AddToCart? MVC▸

contextPath là đường dẫn gốc của web app trên server. Nếu app deploy tại /shop, contextPath = /shop. /AddToCart cứng = http://host/AddToCart (từ root). AddToCart relative = phụ thuộc URL hiện tại. ${contextPath}/AddToCart = http://host/shop/AddToCart — đúng mọi trường hợp.

56 Cart.jsp dùng cart.entrySet() để duyệt giỏ hàng. entrySet() là gì? Tại sao không dùng keySet()? MVC▸

entrySet() trả về tập hợp các Map.Entry chứa cả key và value. keySet() chỉ trả về keys — muốn lấy value phải gọi thêm map.get(key). Dùng entrySet() hiệu quả hơn vì không lookup lại — lấy được cả productId lẫn [product, qty] trong một vòng lặp.

57 Code có <%@page import="java.util.LinkedHashMap"%> nhưng class thực sự khai báo là Map<Integer, Object[]> cart — tại sao import LinkedHashMap nhưng khai báo kiểu Map? JSP/Java▸

Program to interface — khai báo biến kiểu Map (interface) nhưng thực tế là LinkedHashMap (implementation). Import LinkedHashMap để có thể viết new LinkedHashMap<>() trong code. Dùng interface giúp dễ đổi implementation sau mà không cần sửa code dùng biến đó.

Phần 6 — Bảo mật Web
58 Form đăng nhập dùng method="post" — tại sao password không nên gửi bằng method="get"? Bảo mật▸

GET: params xuất hiện trong URL → lưu vào browser history, server access log, proxy log, Referer header. Password sẽ lộ ra như: /login?username=admin&password=123456. POST: params trong request body, không hiện trong URL. Password vẫn nên encrypt (HTTPS) nhưng ít nhất không lộ trong URL.

59 Trang admin kiểm tra quyền bằng "admin".equals(session.getAttribute("role")) trong JSP. Đây có phải cách bảo mật đủ tốt không? Bảo mật▸

Không đủ nếu chỉ kiểm tra trong JSP — JSP kiểm tra ở View layer. User có thể gọi thẳng URL /AdminDashboardServlet bằng cách gõ trực tiếp, bypass JSP. Phải kiểm tra quyền trong Servlet (Controller), và tốt hơn là dùng Filter để intercept mọi request đến admin URL.

Filter pattern: implement Filter, check session trong doFilter(), redirect về login nếu không có quyền.
60 Form tìm kiếm: value="<%= request.getParameter("txtSearch") %>" — đây có thể gây XSS không? Ví dụ user nhập gì? Bảo mật▸

Có thể XSS. Nếu user nhập: "><script>alert(1)</script>, HTML thành: value=""><script>alert(1)</script> → script chạy. Phải escape: value="<%= request.getParameter("txtSearch").replace("<","<").replace(">",">").replace("\"","&quot;") %>" hoặc dùng JSTL <c:out> tự escape.

OWASP: XSS là lỗ hổng phổ biến top 3 trong web application.
61 User có thể thêm sản phẩm vào giỏ bằng cách gọi URL /AddToCart?productId=999&qty=1000 trực tiếp. Điều gì có thể xảy ra? Bảo mật▸

(1) productId=999 không tồn tại → DAO trả về null → NullPointerException nếu không xử lý. (2) qty=1000 vượt tồn kho → nếu không validate server-side, giỏ hàng lưu qty=1000. (3) Kết hợp với hàm reduceQuantity có kiểm tra tồn kho → checkout sẽ thất bại. Nhưng vẫn cần validate input sớm.

62 Form đăng ký có trường type="add" cho địa chỉ — đây là bug gì? Có ảnh hưởng bảo mật không? Bảo mật▸

type="add" không tồn tại — browser fallback về type="text". Không ảnh hưởng bảo mật nhưng là code bẩn. Quan trọng hơn: trường địa chỉ không có validation bất kỳ — user có thể nhập HTML/script vào địa chỉ, lưu xuống DB, khi admin xem có thể bị Stored XSS.

63 CSRF (Cross-Site Request Forgery) là gì? Form ClearCart có dễ bị tấn công CSRF không? Bảo mật▸

CSRF: trang độc hại tạo form giả gửi POST đến /ClearCart khi user đang đăng nhập → giỏ hàng bị xóa mà user không biết. Form hiện tại không có CSRF token → dễ bị tấn công. Phòng chống: thêm hidden input <input type="hidden" name="_csrf" value="${csrfToken}">, Servlet verify token.

64 Checkout form: <form action="${pageContext.request.contextPath}/checkout.jsp" method="post"> — action trỏ thẳng vào file JSP. Vấn đề gì? Bảo mật▸

Action nên trỏ vào Servlet, không phải JSP. Checkout cần: kiểm tra đăng nhập, validate giỏ hàng, trừ tồn kho, tạo đơn hàng trong DB, gửi email. JSP không nên làm những việc này. Trỏ vào JSP trực tiếp bypass toàn bộ logic nghiệp vụ.

Phần 7 — Database & DAO
65 DAO (Data Access Object) pattern là gì? Tại sao cần tách DAO ra khỏi Servlet? Database▸

DAO là lớp trung gian giữa logic nghiệp vụ và database — chỉ chứa các method CRUD (getAll, getById, insert, update, delete). Tách DAO để: (1) Dễ thay đổi DB (MySQL → PostgreSQL) không ảnh hưởng Servlet. (2) Dễ test riêng. (3) Không lặp code connection/query ở nhiều Servlet.

66 Code dùng PreparedStatement thay vì Statement để query DB. Lợi ích chính là gì? Database▸

(1) SQL Injection prevention: params được truyền riêng, DB treat chúng là data, không phải SQL. (2) Performance: câu SQL được compile sẵn, tái sử dụng execution plan. (3) Type safety: setInt, setString đảm bảo kiểu dữ liệu đúng. Không dùng PreparedStatement là lỗ hổng bảo mật nghiêm trọng.

67 Hàm reduceQuantity(productId, qty) dùng WHERE quantity >= ? — tại sao không SELECT kiểm tra trước rồi mới UPDATE? Database▸

SELECT rồi UPDATE tạo ra khoảng thời gian giữa 2 thao tác — race condition: 2 user cùng mua có thể cùng pass SELECT rồi cả 2 UPDATE → oversell. Gộp điều kiện vào WHERE của UPDATE là atomic operation — DB lock dòng khi UPDATE, tránh race condition. Đây là optimistic locking.

68 try (Connection con = new dbConnect().getConnect(); PreparedStatement ps = ...) — cú pháp try(...) này gọi là gì? Lợi ích? Database▸

Try-with-resources (Java 7+). Tự động gọi close() trên Connection và PreparedStatement khi block try kết thúc, dù có exception hay không. Không cần viết finally để close. Tránh connection leak — nếu quên close connection thủ công, DB sẽ hết pool.

69 Class dbConnect viết thường thay vì DbConnect — điều này vi phạm convention nào của Java? Ảnh hưởng gì không? JSP/Java▸

Java convention: tên class dùng PascalCase (UpperCamelCase). dbConnect → nên là DbConnect. Tương tự products → nên là Products, products_DAO → ProductsDAO. Không ảnh hưởng chức năng nhưng vi phạm coding standard, giảm readability, có thể bị điểm trừ khi chấm code style.

70 Nếu DB bị down khi user thêm vào giỏ hàng, code xử lý thế nào? User thấy gì? Database▸

DAO bắt Exception e, gọi e.printStackTrace() (log ra console), trả về null hoặc false. Servlet nhận kết quả → redirect về trang với thông báo lỗi (nếu có xử lý). Nếu không xử lý null kỹ → NullPointerException → trang 500 Internal Server Error. Cần catch và hiển thị thông báo thân thiện.

71 Danh mục sản phẩm lấy từ DB hay hardcode? Dropdown nav và sidebar danh mục có dùng cùng data source không? Database▸

Lấy từ DB qua Servlet → request.setAttribute("categories", categories). Cả dropdown nav và sidebar đều dùng request.getAttribute("categories"). Tuy nhiên cart.jsp có fallback hardcode: String[] defaultCats = {"Phòng khách", "Phòng ngủ", ...} nếu categories null — home_page.jsp không có fallback này.

72 Tại sao nên dùng Connection Pool (như HikariCP, DBCP) thay vì tạo new Connection() mỗi lần query? Database▸

Tạo connection là operation tốn kém (TCP handshake, authentication). Mỗi request tạo mới và đóng → chậm, tốn resource. Connection pool duy trì pool sẵn các connection, tái sử dụng. Project hiện tại tạo connection mới mỗi lần (new dbConnect().getConnect()) — không hiệu quả cho production.

81 Project có chức năng chatbot gọi /ChatServlet. Theo bạn, ChatServlet có thể gọi API AI ngoài (như OpenAI) không? Cần xử lý gì thêm? Kiến trúc▸

Hoàn toàn có thể. ChatServlet gọi HTTP request đến OpenAI API, nhận response, trả text về client. Cần xử lý thêm: (1) API key bảo mật (lưu trong environment variable, không hardcode). 82 Nav menu được copy-paste vào tất cả JSP. Nếu cần đổi menu, phải sửa bao nhiêu file? Giải pháp tốt hơn là gì? Kiến trúc▸

Ít nhất 4 file (home, cart, product_detail, lienhe). Giải pháp: JSP Include — tách nav thành nav.jsp, các trang dùng <%@ include file="nav.jsp" %> (static include, compile time) hoặc <jsp:include page="nav.jsp"/> (dynamic include, runtime). Thay đổi 1 file → áp dụng cho toàn bộ.

84 Nếu 1000 user cùng thêm vào giỏ cùng lúc, giỏ hàng lưu trong session — server có vấn đề gì không? Kiến trúc▸

HTTP: dữ liệu truyền plaintext — password, thông tin đơn hàng, thẻ tín dụng có thể bị nghe lén (Man-in-the-Middle). HTTPS mã hóa TLS. Với HTTP, JSESSIONID cookie cũng không được bảo vệ — attacker trên cùng mạng có thể hijack session. Google Chrome đánh dấu HTTP là "Not Secure".

86 Trang liên hệ (lienhe.jsp) có form gửi tin nhắn nhưng action trỏ về đâu? Dữ liệu có thực sự được lưu không? Bẫy▸

87 Phân biệt request.getParameter() và request.getAttribute() — khi nào dùng cái nào? MVC▸

getParameter(): lấy data từ HTTP request (form field, query string) — chỉ đọc được String. getAttribute(): lấy object Java được set bằng setAttribute() trong cùng request — có thể là bất kỳ kiểu nào (List, Object...). Servlet set product list bằng setAttribute, JSP đọc bằng getAttribute. User form data dùng getParameter.

88 Tại sao code dùng try { Integer.parseInt(request.getParameter("productId")) } catch khi lấy productId? Không parse thẳng được sao? JSP/Java▸

getParameter() luôn trả về String (hoặc null). parseInt ném NumberFormatException nếu String không phải số (null, rỗng, chữ). User có thể manipulate URL: ?productId=abc → exception → 500 error. Try-catch bắt và xử lý gracefully (redirect về 404 thân thiện).

89 Nếu project cần thêm tính năng "Đánh giá sản phẩm (rating)", cần thêm gì vào MVC? Kiến trúc▸

Model: class Review (userId, productId, rating, comment, date). DAO: ReviewDAO (addReview, getByProductId, getAvgRating). Servlet: AddReviewServlet (POST), ProductDetailServlet load thêm reviews. View: thêm section đánh giá trong product_detail.jsp, form gửi review, hiển thị list reviews. DB thêm bảng reviews.

90 Project không có trang 404.html hay error page tùy chỉnh. Khi user truy cập URL không tồn tại, thấy gì? Cách thêm error page? Kiến trúc▸

Thấy trang lỗi mặc định của Tomcat — xấu và lộ thông tin stack trace. Cấu hình trong web.xml: <error-page><error-code>404</error-code><location>/404.jsp</location></error-page>. Tương tự cho 500. Custom error page ẩn thông tin kỹ thuật và tạo UX nhất quán.

Phần 10 — 10 Câu Cuối: Câu hỏi tổng kết
91 Vòng lặp tính subtotal: subtotal += p.getPrice() * qty — tại sao dùng double thay vì int hay long? JSP/Java▸

Giá có thể là decimal (1.500.000,5 đồng — dù hiếm). int max ~2.1 tỷ đủ dùng, nhưng nhân giá × số lượng lớn có thể overflow. long tốt hơn int cho tiền tệ. Thực tế, tiền tệ nên dùng BigDecimal để tránh floating-point precision error (0.1 + 0.2 ≠ 0.3 trong double). Dùng double là đơn giản nhưng không chính xác 100%.

92 Khi user mở 2 tab trình duyệt cùng lúc và thêm hàng từ cả 2 tab — giỏ hàng có bị conflict không? Session▸

Cùng browser = cùng session (cùng JSESSIONID cookie). Mỗi request AddToCart là synchronous — nếu cùng lúc exact có thể race condition ở code Java (Map không thread-safe). Nên dùng synchronized(session) { ... } hoặc ConcurrentHashMap khi modify cart trong session để tránh data corruption.

93 Tại sao không nên để toàn bộ JavaScript inline trong JSP mà nên tách ra file .js riêng? JavaScript▸

(1) Caching: file .js riêng được browser cache, không tải lại mỗi request. Inline JS tải lại mỗi lần. (2) Maintainability: một logic ở một chỗ, không lặp lại ở nhiều JSP. (3) Separation of concerns: JS logic tách khỏi HTML. Trường hợp cần JSP variables trong JS (như contextPath) thì vẫn cần inline một phần.

94 Mật khẩu trong DB được lưu dạng gì? Plain text hay hash? Nếu plain text thì rủi ro gì khi DB bị hack? Bảo mật▸

Không nhìn thấy code hash trong project → có thể lưu plain text. Nếu plain text: DB bị hack → toàn bộ password user lộ ngay. Phải hash với bcrypt hoặc Argon2 — thuật toán một chiều có salt, không thể reverse. MD5/SHA1 không đủ vì dễ rainbow table. Đây là yêu cầu bắt buộc cho mọi ứng dụng thực tế.

95 Tên Servlet: AddToCart, RemoveFromCart, ClearCart, UpdateCart — đây là thiết kế URL theo style gì? RESTful không? MVC▸

Không RESTful — RESTful dùng HTTP method để phân biệt action: POST /cart (add), DELETE /cart/{id} (remove), PUT /cart/{id} (update), DELETE /cart (clear). Project dùng nhiều URL riêng, đều dùng POST/GET — style RPC truyền thống, phổ biến với Servlet nhưng không RESTful.

96 Giả sử giảng viên hỏi: "Chức năng nào trong project bạn thấy chưa hoàn thiện nhất?" — bạn trả lời gì? Bẫy▸

Trả lời thành thật và phân tích kỹ: (1) Coupon chỉ là UI giả — không thực sự giảm tiền. (2) Checkout trỏ vào JSP, không có Servlet xử lý thanh toán thực. (3) Mật khẩu có thể lưu plain text. (4) Thiếu responsive. (5) Không có validation server-side đầy đủ. Trả lời thẳng thắn + nêu cách sửa sẽ được đánh giá cao hơn che giấu.

97 Nếu project cần deploy lên internet (VPS/cloud), cần chuẩn bị thêm những gì so với chạy localhost? Kiến trúc▸

(1) HTTPS certificate (Let's Encrypt free). (2) Database config production (không dùng localhost). (3) Đổi mật khẩu DB, xóa default credentials. (4) Cấu hình firewall, chỉ mở port 80/443. (5) Log rotation. (6) Backup DB. (7) Sửa lỗi backslash trong CSS path. (8) Env variable cho secrets thay vì hardcode. (9) Error page tùy chỉnh.

98 Class tên products (số nhiều, chữ thường) đại diện cho một sản phẩm — tên đặt có đúng convention không? Nên đặt tên thế nào? JSP/Java▸

Không đúng 2 điểm: (1) Class Java dùng PascalCase → phải là Products. (2) Class đại diện 1 object → nên số ít: Product. Vậy tên đúng là Product. Tương tự products_DAO → ProductDAO. Naming convention quan trọng vì code đọc như văn xuôi: Product p = new Product() rõ hơn products p = new products().

99 Nếu user A và user B đăng nhập cùng server, session của họ có cùng nhau không? Làm sao server biết request nào của ai? Session▸

Mỗi user có session riêng biệt, nhận diện qua JSESSIONID cookie khác nhau. Browser A gửi Cookie: JSESSIONID=ABC123, browser B gửi JSESSIONID=XYZ789. Tomcat dùng ID này để lookup SessionMap và trả về đúng session. Session hoàn toàn isolated giữa các user.
